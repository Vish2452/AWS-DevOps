# KMS — Key Management Service

> Centralized key management for encryption across AWS services. The "key locker" that protects all your sensitive data.

---

## Real-World Analogy

KMS is like a **bank vault with safety deposit boxes**:
- **KMS Master Key** = the vault itself (never leaves the bank)
- **Data Key** = a copy of the key to your safety deposit box
- **Envelope Encryption** = the vault key protects the box key, the box key protects your valuables
- You can't take the vault key home (KMS key never leaves AWS)
- The bank logs every time someone accesses the vault (CloudTrail audit)

---

## Key Types

| Type | Management | Rotation | Cost | Use Case | Real-World Example |
|------|-----------|----------|------|----------|-------------------|
| **AWS Managed** | AWS | Auto (yearly) | Free usage | Default service encryption | S3 default encryption (aws/s3 key) |
| **Customer Managed (CMK)** | You | Configurable | $1/month + usage | Custom encryption requirements | Encrypt production database with your own key |
| **AWS Owned** | AWS | Internal | Free | Shared across accounts | Internal AWS service encryption |
| **External (imported)** | You | Manual | $1/month + usage | Compliance requiring specific key material | Bring-your-own-key for regulatory compliance |

## Envelope Encryption — How It Actually Works

```
Step 1: You ask KMS to create a Data Key
        KMS → returns {plaintext_data_key, encrypted_data_key}

Step 2: Your application uses plaintext_data_key to encrypt your data
        Data + plaintext_data_key → encrypted_data

Step 3: Store the encrypted data + encrypted_data_key together
        (Immediately discard the plaintext_data_key from memory!)

Step 4: To decrypt later:
        encrypted_data_key → KMS → plaintext_data_key
        encrypted_data + plaintext_data_key → original_data

WHY this design?
- KMS has a 4KB encryption limit — can't encrypt large files
- Envelope encryption lets you encrypt files of ANY size
- The master key NEVER leaves KMS (FIPS 140-2 validated hardware)
- Only the small data key needs KMS — fast and cheap
```

```
┌─────────────────────────────────────────────────────────┐
│                     KMS (Vault)                          │
│  ┌───────────────────────────────┐                      │
│  │  Master Key (CMK)             │ ← Never leaves KMS!  │
│  │  Encrypts/decrypts data keys  │                      │
│  └───────────────────────────────┘                      │
└────────────────────┬────────────────────────────────────┘
                     │
            ┌────────┴────────┐
            │  Data Key        │ ← Generated by KMS
            │  (plaintext)     │ ← Used to encrypt YOUR data
            └────────┬────────┘
                     │
            ┌────────┴────────────────────┐
            │  Your Data                   │
            │  (encrypted with data key)   │
            │  + Encrypted Data Key        │ ← Stored together
            └─────────────────────────────┘
```

## Real-Time Example 1: S3 Bucket Encryption

**Scenario:** Your company stores customer data in S3. Compliance requires all data to be encrypted at rest with keys you control and can audit.

```bash
# Create CMK for S3 encryption
KEY_ID=$(aws kms create-key \
    --description "Customer data encryption key" \
    --key-usage ENCRYPT_DECRYPT \
    --query 'KeyMetadata.KeyId' --output text)

aws kms create-alias --alias-name alias/customer-data-key --target-key-id $KEY_ID

# Enable automatic rotation (yearly)
aws kms enable-key-rotation --key-id $KEY_ID

# Set S3 bucket to use this key by default
aws s3api put-bucket-encryption \
    --bucket customer-data-bucket \
    --server-side-encryption-configuration '{
        "Rules": [{
            "ApplyServerSideEncryptionByDefault": {
                "SSEAlgorithm": "aws:kms",
                "KMSMasterKeyID": "alias/customer-data-key"
            },
            "BucketKeyEnabled": true
        }]
    }'
```

**Explanation:** Every file uploaded to this bucket is automatically encrypted using your CMK. If auditors ask "Who accessed the encryption key?", you can show CloudTrail logs. If you suspect a breach, you can disable the key — making ALL data unreadable instantly.

## Real-Time Example 2: Cross-Account Key Sharing

**Scenario:** Your organization has Dev (111111111111) and Prod (222222222222) accounts. Prod encrypts AMIs. Dev needs to copy and decrypt them for testing.

```json
// Key policy that grants cross-account access
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "Allow Prod Account Full Access",
            "Effect": "Allow",
            "Principal": {"AWS": "arn:aws:iam::222222222222:root"},
            "Action": "kms:*",
            "Resource": "*"
        },
        {
            "Sid": "Allow Dev Account to Decrypt",
            "Effect": "Allow",
            "Principal": {"AWS": "arn:aws:iam::111111111111:role/DevOpsRole"},
            "Action": [
                "kms:Decrypt",
                "kms:DescribeKey",
                "kms:CreateGrant",
                "kms:ReEncryptFrom"
            ],
            "Resource": "*"
        }
    ]
}
```

**Explanation:** In enterprise environments, you often need to share encrypted resources across accounts. KMS key policies control exactly who can use which operations. Dev can decrypt but NOT encrypt new data with the production key.

## Real-Time Example 3: Emergency Key Disable (Security Incident)

**Scenario:** You discover a data breach. Customer data may be compromised. You need to make all data unreadable IMMEDIATELY.

```bash
# EMERGENCY: Disable the encryption key
# All data encrypted with this key becomes UNREADABLE instantly
aws kms disable-key --key-id alias/customer-data-key

# Once you've contained the breach, re-enable
aws kms enable-key --key-id alias/customer-data-key

# Or schedule deletion (irreversible after waiting period!)
aws kms schedule-key-deletion --key-id $KEY_ID --pending-window-in-days 7
# WARNING: After 7 days, ALL data encrypted with this key is PERMANENTLY lost
```

**Explanation:** This is a powerful security control. Rather than deleting terabytes of data (which takes time), disabling the key makes ALL data across ALL services instantly unreadable. S3, EBS, RDS — everything encrypted with this key becomes inaccessible in seconds.

| **SNS** | Topic encryption | Published messages encrypted |

## Key Policies vs IAM Policies

```
Key Policy (on the KMS key):
"Who can USE this key?"
→ Account 111111111111 can encrypt/decrypt
→ Role DevOpsAdmin can manage the key
→ User Alice can only view key details

IAM Policy (on the user/role):
"What keys can this user access?"
→ Alice can use keys with alias/dev-*
→ DevOpsAdmin can use ALL keys in the account

BOTH must allow access for the action to succeed!
Key Policy = resource-level permission
IAM Policy = identity-level permission
```

## Labs

### Lab 1: Create and Use CMK
```bash
# Create CMK
KEY_ID=$(aws kms create-key \
    --description "Application encryption key" \
    --key-usage ENCRYPT_DECRYPT \
    --query 'KeyMetadata.KeyId' --output text)

aws kms create-alias --alias-name alias/app-key --target-key-id $KEY_ID

# Encrypt data (< 4KB)
aws kms encrypt --key-id alias/app-key \
    --plaintext fileb://secret.txt \
    --output text --query CiphertextBlob | base64 --decode > secret.encrypted

# Decrypt data
aws kms decrypt --ciphertext-blob fileb://secret.encrypted \
    --output text --query Plaintext | base64 --decode > secret.decrypted

# Enable auto-rotation
aws kms enable-key-rotation --key-id $KEY_ID
```

### Lab 2: Envelope Encryption for Large Files
```bash
# Generate data key
DATA_KEY=$(aws kms generate-data-key --key-id alias/app-key \
    --key-spec AES_256 --query '{Plaintext: Plaintext, CiphertextBlob: CiphertextBlob}')

# Use the plaintext data key to encrypt a large file with OpenSSL
echo $DATA_KEY | jq -r '.Plaintext' | base64 --decode > /tmp/datakey.bin
openssl enc -aes-256-cbc -in largefile.zip -out largefile.encrypted -pass file:/tmp/datakey.bin

# Store encrypted file + encrypted data key together
echo $DATA_KEY | jq -r '.CiphertextBlob' > largefile.encrypted.key

# Clean up plaintext key
rm /tmp/datakey.bin
```

### Lab 3: Audit Key Usage
```bash
# See who used the key in the last 7 days
aws cloudtrail lookup-events \
    --lookup-attributes AttributeKey=ResourceType,AttributeValue=AWS::KMS::Key \
    --start-time "2026-02-23T00:00:00Z" \
    --end-time "2026-03-02T23:59:59Z" \
    --max-results 20
```

## Interview Questions

1. **What is KMS and why is it important?**
   > KMS is a managed encryption key service. It provides centralized key management, automatic key rotation, and full audit logging. Required for compliance (HIPAA, PCI, SOC2). Keys are stored in FIPS 140-2 validated hardware.

2. **What is envelope encryption and why does KMS use it?**
   > KMS master key encrypts a data key, data key encrypts the actual data. KMS can only encrypt up to 4KB directly. Envelope encryption lets you encrypt data of any size. The master key never leaves KMS hardware — maximum security.

3. **What is the difference between AWS Managed and Customer Managed keys?**
   > AWS Managed: created by AWS for specific services, auto-rotated yearly, free, you can't manage policies. Customer Managed: you create, you control key policy, configurable rotation, $1/month, full audit control. Use CMK for production compliance.

4. **How does KMS key rotation work?**
   > AWS auto-rotates (new key material yearly), but keeps old versions for decrypting existing data. The key ID and alias don't change — your app doesn't need updates. Data encrypted with old material still decryptable. Manual rotation = create new key and update references.

5. **How would you handle a security incident where data may be compromised?**
   > Immediately disable the KMS key — all data encrypted with it becomes unreadable globally. Investigate the breach. If contained, re-enable the key. If the key itself is compromised, schedule deletion and re-encrypt data with a new key.

6. **What is a key policy and how is it different from IAM policy?**
   > Key policy: attached to the KMS key, defines who can use it (resource-level). IAM policy: attached to users/roles, defines what they can access (identity-level). Both must allow the action. Key policy is the "final authority" — if it denies, IAM policy can't override.

7. **How does S3 SSE-KMS differ from SSE-S3?**
   > SSE-S3: AWS manages everything, one key for all objects, no audit trail per object. SSE-KMS: You choose the key, each object gets a unique data key, CloudTrail logs every encryption/decryption. Use SSE-KMS for compliance.

8. **Can you encrypt existing unencrypted resources?**
   > Depends on service. S3: copy object with encryption. EBS: snapshot → copy with encryption → create volume. RDS: snapshot → restore with encryption. You cannot encrypt in-place — must create encrypted copy.
